## 1. 프로세스 개요

### 1.1 정의

**"사용자 경험 기반 기획 검증 프로세스"**는 프로덕트 개발 착수 전, 기획 문서의 완성도를 높이기 위해 **PM/PO와 기획 담당자가 실제 사용자 경험을 시뮬레이션하며 세부 플로우를 하나씩 검증하는 협업 방법론**입니다.

### 1.2 핵심 특징

```
기존 기획 검토 방식:
├─ 문서 읽기 → 코멘트 남기기
├─ 추상적 피드백 ("더 직관적으로")
└─ 기술 스펙과 괴리

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

새로운 방식 (UX 기반 검증):
├─ 사용자 시점에서 경험 시뮬레이션
├─ 구체적 화면/행동/반응 묘사
├─ 즉시 기술 스펙 연동 수정
└─ 실제 개발 가능한 수준까지 정교화
```

### 1.3 적용 시점

```
프로젝트 타임라인:

[아이디어] → [시장조사] → [초기기획] → [🎯 여기!] → [개발] → [출시]
                                          ↑
                                    UX 기반 검증
                                    (2-3주 소요)
```

## 2. 왜 필요한가

### 2.1 기존 기획 방식의 문제점

#### 문제 1: 추상적 기획서

```
기획서: "사용자가 직관적으로 이용할 수 있어야 함"

개발자: "직관적이란 게 구체적으로 뭔가요?"
       "버튼이 몇 개? 어디에? 뭐라고 써있나요?"

→ 결과: 개발 중 계속 질문, 일정 지연
```

#### 문제 2: 기술-기획 간 괴리

```
기획: "실시간으로 알림을 보여줍니다"

개발 후:
├─ 알림이 너무 많이 떠서 방해됨
├─ 타이밍이 어색함 (대화 중간에 끼어듦)
└─ 닫기 버튼이 없어서 계속 쌓임

→ 결과: 재작업, 리소스 낭비
```

#### 문제 3: 엣지 케이스 미고려

```
기획: "검색 결과를 표시합니다"

실제 개발 시:
├─ 검색 결과가 0개일 때는?
├─ 검색어가 너무 길 때는?
├─ 검색 중 에러 나면?
└─ 검색 결과가 1000개면?

→ 결과: 개발 중 매번 PM에게 질문
```

### 2.2 UX 기반 검증의 효과

```
✅ 개발 전 모든 시나리오 사전 검증
✅ 개발자 질문 90% 감소
✅ 재작업 70% 감소
✅ 일정 준수율 향상
✅ 프로덕트 품질 향상
```

### 2.3 ROI 분석

```
투입:
├─ PM 시간: 10-15시간 (2주)
├─ 기획자 시간: 20-30시간 (2주)
└─ 총: 30-45시간

절감:
├─ 개발 중 재작업: -100시간
├─ 출시 후 수정: -200시간
├─ 고객 불만 대응: -50시간
└─ 총: 350시간

ROI: 7.7배 (350 / 45)
```

## 3. 프로세스 구조

### 3.1 전체 흐름

```
┌─────────────────────────────────────────────┐
│  Phase 1: 준비 (1-2일)                       │
├─────────────────────────────────────────────┤
│  • 기존 기획 문서 정리                        │
│  • 사용자 플로우 대분류 정의                  │
│  • 우선순위 설정                             │
│  • 역할 분담                                 │
└─────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────┐
│  Phase 2: 검증 (1-2주)                       │
├─────────────────────────────────────────────┤
│  • 파트별 상세 UX 제시                       │
│  • PM 피드백 수렴                            │
│  • 즉시 수정 및 문서 업데이트                │
│  • 다음 파트 진행                            │
└─────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────┐
│  Phase 3: 정리 (2-3일)                       │
├─────────────────────────────────────────────┤
│  • 최종 문서 통합                            │
│  • 개발팀 인수인계 자료 작성                 │
│  • QA 체크리스트 생성                        │
│  • 킥오프 미팅                               │
└─────────────────────────────────────────────┘
```

### 3.2 세부 단계

```
Step 1: 파트 정의
├─ 전체 사용자 여정을 10-20개 파트로 분할
├─ 각 파트의 시작/끝 명확히 정의
└─ 우선순위 부여 (Critical > High > Medium > Low)

Step 2: 파트별 시뮬레이션
├─ 기획자가 상세 UX 시나리오 작성
├─ PM이 사용자 입장에서 검토
└─ 피드백 수렴

Step 3: 즉시 반영
├─ 피드백을 바로 문서에 반영
├─ 연관 기술 스펙 동시 업데이트
└─ 다음 파트로 진행

Step 4: 최종 검증
├─ 전체 플로우 통합 시뮬레이션
├─ 엣지 케이스 추가 검증
└─ 개발 인수인계
```

## 4. 역할 정의

### 4.1 PM/PO 역할

**책임:**
- 최종 의사결정권자
- 사용자 관점 검증
- 비즈니스 요구사항 확인

**활동:**
```
1. 각 파트 시나리오 읽기
   └─ "실제로 이렇게 쓸 것 같은가?"

2. 피드백 제공
   └─ 구체적으로: "이 버튼은 여기가 아니라 저기에"
   └─ 이유 설명: "왜냐하면 사용자는..."

3. 우선순위 조정
   └─ "이 기능은 빼고, 저건 꼭 넣자"

4. 최종 승인
   └─ "OK, 이대로 개발 진행"
```

**필요 역량:**
- 사용자 공감 능력
- 구체적 피드백 능력
- 빠른 의사결정

### 4.2 기획 담당자 역할

**책임:**
- 상세 UX 시나리오 작성
- 기술 스펙 연동
- 문서 업데이트

**활동:**
```
1. 파트별 상세 시나리오 작성
   ├─ 사용자가 보는 화면 (구체적)
   ├─ 사용자가 하는 행동
   ├─ 시스템 반응
   └─ 관련 기술 스펙

2. PM 피드백 수렴
   └─ 명확히 이해, 추가 질문

3. 즉시 반영
   ├─ 해당 파트 수정
   ├─ 관련 문서 업데이트 (FE, BE, AI)
   └─ 영향받는 다른 파트 확인

4. 문서 관리
   └─ 버전 관리, 변경 이력 추적
```

**필요 역량:**
- 세밀한 UX 묘사 능력
- 기술 이해도 (FE, BE, AI)
- 빠른 수정 능력
- 문서 작성 능력

## 5. 진행 방법

### 5.1 미팅 방식

#### Option A: 동기 방식 (실시간 미팅)

```
시간: 매일 2시간 (오전 또는 오후)
장소: 회의실 또는 Zoom

진행:
├─ 기획자가 화면 공유
├─ 파트 시나리오 함께 읽기
├─ PM이 즉시 피드백
├─ 기획자가 바로 수정
└─ 다음 파트로

장점: 빠른 피드백, 실시간 논의
단점: 시간 맞추기 어려움
```

#### Option B: 비동기 방식 (문서 기반)

```
도구: Notion, Google Docs, Slack

진행:
├─ 기획자가 파트별 문서 작성
├─ PM이 시간될 때 검토 + 코멘트
├─ 기획자가 확인 후 수정
└─ PM이 재검토 후 승인

장점: 시간 유연, 깊은 사고 가능
단점: 속도 느림, 맥락 전달 어려움
```

#### Option C: 하이브리드 (추천)

```
비동기 + 동기 혼합:

1. 기획자가 사전에 문서 작성
2. PM이 미리 읽고 질문/코멘트 남김
3. 실시간 미팅에서 집중 논의
4. 즉시 수정 후 다음 파트

장점: 효율성 + 깊이
```

### 5.2 일정 계획

#### 2주 집중 방식 (추천)

```
Week 1:
├─ 월: 준비 + 온보딩 파트 (3개)
├─ 화: 핵심 플로우 1 (미팅 전 준비, 3개)
├─ 수: 핵심 플로우 2 (미팅 중, 4개)
├─ 목: 핵심 플로우 3 (미팅 후, 3개)
└─ 금: 부가 기능 (검색, 설정, 2개)

Week 2:
├─ 월: 엣지 케이스 (에러, 오프라인, 3개)
├─ 화: 통합 시뮬레이션 (전체 플로우)
├─ 수: 개선 및 폴리싱
├─ 목: 최종 문서 정리
└─ 금: 개발팀 인수인계

총: 15-20개 파트, 10일
```

## 6. 파트 분류 방법

### 6.1 분류 기준

```
대분류: 사용자 여정 단계
├─ 온보딩 (처음 사용)
├─ 핵심 기능 사용 (메인 플로우)
├─ 부가 기능 (검색, 설정 등)
└─ 예외 상황 (에러, 오프라인 등)

소분류: 화면 전환 기준
├─ 화면 A → 화면 B = 1개 파트
├─ 사용자 행동 완결 단위
└─ 평균 1-3분 내 완료 가능
```

### 6.2 파트 크기 가이드

```
✅ 적절한 크기:
"로그인 화면에서 이메일/비밀번호 입력 후 로그인 버튼 클릭,
성공 시 대시보드로 이동, 실패 시 에러 메시지 표시"

❌ 너무 큰 크기:
"사용자가 서비스를 이용하는 전체 플로우"
→ 10-20개 파트로 분할 필요

❌ 너무 작은 크기:
"로그인 버튼에 마우스 올리면 색상 변경"
→ 너무 세세함, 디자인 QA 수준
```

### 6.3 우선순위 부여

```
Critical (반드시 검증):
├─ 온보딩 (첫인상)
├─ 핵심 가치 전달 순간
├─ 결제/중요 의사결정
└─ 데이터 손실 가능성 있는 부분

High (중요):
├─ 자주 사용되는 기능
├─ 복잡한 플로우
└─ 다른 파트에 영향 큰 부분

Medium (선택):
├─ 부가 기능
└─ 사용 빈도 낮음

Low (생략 가능):
├─ 설정 화면
└─ 도움말
```

## 8. 피드백 처리

### 8.1 피드백 유형

#### Type 1: 승인 ✅

```
PM: "좋습니다, 다음으로"
PM: "OK"
PM: "컨펌"

→ 기획자 액션:
   - 다음 파트로 진행
   - 체크리스트에 ✅ 표시
```

#### Type 2: 수정 요청 💬

```
PM: "이 버튼은 여기가 아니라 우측 상단에"
PM: "문구를 '시작하기'에서 '지금 시작'으로"
PM: "색상이 너무 화려함, 톤 다운"

→ 기획자 액션:
   1. 즉시 해당 파트 수정
   2. 관련 기술 스펙 업데이트
   3. 영향받는 다른 파트 확인
   4. PM에게 재제시
```

#### Type 3: 질문/추가 정보 요청 ❓

```
PM: "이 경우는 어떻게 되나요?"
PM: "오프라인일 때는?"
PM: "데이터가 없으면?"

→ 기획자 액션:
   1. 엣지 케이스 추가
   2. 상세 시나리오 작성
   3. 재제시
```

#### Type 4: 기능 추가/삭제 🔄

```
PM: "이 기능은 MVP에서 빼자"
PM: "여기에 알림 추가 필요"

→ 기획자 액션:
   1. 전체 플로우 재검토
   2. 우선순위 재조정
   3. 관련 모든 문서 업데이트
   4. 일정 영향 분석

## 9. 문서 연동

### 9.1 연동 대상 문서

```
피드백 발생 시 업데이트해야 할 문서들:

1. UX 플로우 문서
   └─ 해당 파트 시나리오 수정

2. 프론트엔드 기술 스펙
   ├─ 컴포넌트 구조 수정
   ├─ 상태 관리 수정
   └─ API 호출 수정

3. 백엔드 API 설계
   ├─ 엔드포인트 추가/수정
   ├─ Request/Response 수정
   └─ DB 스키마 수정 (필요 시)

4. AI 프롬프트 (해당 시)
   └─ 프롬프트 수정

5. 디자인 스펙
   └─ Figma 파일 업데이트 필요 표시

6. 개발 일정
   └─ 공수 재산정
```

### 9.2 업데이트 프로세스

```
Step 1: 영향 범위 파악
├─ 이 변경이 어디에 영향을 주는가?
├─ FE only? BE도? DB도?
└─ 다른 파트에도 영향 있는가?

Step 2: 우선순위 업데이트
├─ UX 플로우 문서 (항상)
├─ 기술 스펙 (영향 있는 부분만)
└─ 개발 일정 (큰 변경 시)

Step 3: 변경 이력 기록
└─ Git commit 또는 문서 버전 관리

Step 4: PM에게 확인
└─ "이렇게 수정했습니다" 재제시
```

### 9.3 버전 관리

```
문서 파일명:
├─ 05_프론트엔드_기술_스펙_v1.0.md
├─ 05_프론트엔드_기술_스펙_v1.1.md (피드백 반영)

├─ git commit -m "UX 피드백 반영: 온보딩 이메일 인증 추가"

## 15. FAQ

### Q1. 이미 개발 시작했는데 적용 가능한가요?

```
A: 가능하지만 비효율적입니다.

권장:
├─ 개발 중단하고 검증 먼저 (2주)
├─ 재작업 비용 < 검증 비용
└─ 장기적으로 더 빠름

차선:
├─ 미개발 기능만 검증
└─ 개발 완료 기능은 QA 강화
```

### Q2. 디자이너는 언제 참여하나요?

```
A: 검증과 동시에 진행 가능

옵션 1: 검증 후 디자인
├─ 검증 완료 → Figma 디자인
└─ 확실하지만 느림

옵션 2: 병렬 진행 (추천)
├─ 검증하면서 디자이너도 스케치
├─ 피드백 반영하며 정교화
└─ 빠르고 효율적
```

### Q3. 모든 프로젝트에 필요한가요?

```
A: 프로젝트 규모에 따라

필수:
├─ 복잡한 프로덕트 (SaaS, 플랫폼)
├─ 사용자 이탈 비용 큼 (커머스)
└─ 재작업 비용 큼 (대규모 팀)

선택:
├─ 간단한 랜딩 페이지
├─ MVP 실험 (빠른 검증 우선)
└─ 이미 유사 경험 풍부한 팀
```

### Q4. PM이 기술을 몰라도 되나요?

```
A: 기본 이해는 필요합니다.

필요한 수준:
├─ 프론트엔드: "버튼 누르면 API 호출한다"
├─ 백엔드: "데이터베이스에 저장한다"
├─ AI: "프롬프트로 결과 나온다"

불필요한 수준:
├─ React Hooks 사용법
├─ SQL 쿼리 최적화
└─ AI 모델 파인튜닝

→ 기획자가 기술 설명 잘 하면 OK
```
